Advanced Encryption Options

SSE-C (Server-Side Encryption)

R2 supports Server-Side Encryption with Customer-Provided Keys. You supply your own key on every put/get request — Cloudflare encrypts/decrypts server-side but never stores your key. Compatible with the S3 API and Workers Bindings (ssecKey parameter).

Key responsibility is yours:
If you lose your SSE-C key, Cloudflare cannot recover the object body. Store keys in the Secrets Store or an external KMS.

Client-Side Encryption

Encrypt data before upload using the Web Crypto API (available in Workers) or any library. The ciphertext is what gets stored in R2/D1/DO — Cloudflare never sees the plaintext. This is one of the strongest privacy guarantees: zero-knowledge storage.

Best for:
PII, PHI, financial records, or any data where even the storage provider should not have access to content.
Key Management:
Default encryption keys are managed by Cloudflare's internal key management systems. For SSE-C or client-side encryption, you manage your own keys — use the Secrets Store for secure distribution.
Strict TLS/SSL — All data transferred (data-in-transit) between clients, Cloudflare Workers, and backend storage (R2, D1, Durable Objects) is secured using TLS/SSL.

Secure API Access — Access via HTTPS-only APIs or the Wrangler CLI is strictly enforced over HTTPS. Developers can also use Bindings within a Worker to interact with resources in-process — no network hop.

Protocol Enforcement — Plaintext HTTP can be entirely disabled. Enable HSTS to instruct browsers to always use HTTPS. Developers can choose which TLS version and cipher suites to enforce.

Post-Quantum Cryptography (PQC) — All TLS 1.3 connections support X25519MLKEM768 hybrid key agreement, protecting against harvest now, decrypt later attacks. PQC is enabled by default for visitors → Cloudflare, and internal connections are already PQ-secured. Origin connections support PQC too when your origin is configured for it.

Body Buffering Controls — Configuration Rules let you control how Cloudflare buffers request/response bodies. Set to None to stream data directly without inspection (zero-read pass-through), or Full to buffer entirely before forwarding. Useful when compliance requires that Cloudflare does not inspect certain requests.
Role-Based Access Control — Administrators can assign granular roles to developers, with specific permission types across dashboard and API. Use scoped API Tokens with least-privilege permissions for programmatic access.

Encrypted Secrets — Workers allow developers to use encrypted environment variables (Secrets), ensuring sensitive API keys and tokens are never hardcoded in source code. For centralized, account-level secrets management, use the Secrets Store — a dedicated service that encrypts and distributes secrets across all Cloudflare data centers.

Mutual TLS (mTLS) — Workers can be configured to use mTLS to securely authenticate and communicate with backend servers or external APIs.
Cloudflare Privacy Platform:
A suite of enterprise services that let you build privacy-preserving products using IETF standards. Separate user identity from user activity at the network level.

Privacy Proxy

A MASQUE-based forward proxy that separates user identity from user activity. Users authenticate to the proxy without revealing which destinations they visit; destination servers see Cloudflare IPs without learning who made the request. Powers Microsoft Edge Secure Network and acts as a second-hop relay for iCloud Private Relay.

Privacy Gateway

Implements the Oblivious HTTP (OHTTP) IETF standard. A trusted relay forwards encrypted requests between client and server — the relay cannot read application data, and the server cannot see the client’s IP. Ideal for privacy-sensitive APIs, telemetry, and analytics endpoints.

Compliance relevance:
Privacy Proxy and Privacy Gateway help meet GDPR data minimization requirements by ensuring PII (like IP addresses) is never exposed to application backends.
Transparency by default:
Cloudflare publishes transparency data and maintains strong privacy commitments as part of its core business model.
Transparency Report — Published semi-annually. Covers law enforcement requests, national security orders, and government data requests. Cloudflare has never turned over encryption or authentication keys, installed any law enforcement software on its network, provided any government a feed of content transiting the network, or modified content at the request of law enforcement. These are maintained as standing warrant canary commitments.

Privacy-First Business Model — Cloudflare does not sell personal data and does not data-mine user traffic for advertising. Revenue comes from security and performance products, not from monetizing personal information.

Responsible AI — Cloudflare does not train LLMs on customer data. Under the EU AI Act, inference-only workloads like Workers AI do not carry the same obligations as training-phase AI or high-risk AI systems. AI Crawl Control gives site owners transparency and control over how AI bots interact with their content.
Talking point:
Cloudflare’s warrant canary commitments and transparency reports provide verifiable assurance that goes beyond certifications — it demonstrates operational trustworthiness.
